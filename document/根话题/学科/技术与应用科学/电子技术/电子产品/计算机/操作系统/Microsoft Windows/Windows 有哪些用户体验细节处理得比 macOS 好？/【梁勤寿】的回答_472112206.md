 <p data-pid="_iSd2f9C">好嘛，破千赞了，那我就认真吐槽一下。利益相关：macOS 和 Windows 深度用户，平时工作设备是 MacBook Pro 2017 + Surface Pro 4。</p>
 <p data-pid="7RX1-uPP">一个被其他答主说过很多次的事情：窗口管理。具体点来说，一个小问题：屏幕边缘的利用。</p>
 <p data-pid="KdMhLbQy">稍有交互常识的人都知道，传统键鼠操作方式中，屏幕的边缘和角落是最方便的操作位置之一——因为光标移动到屏幕边缘就不会继续移动了。更感性的来讲，就是我“鼠标使劲往一个方向一甩”，就可以“闭着眼睛”达到我的目的，节约了用户宝贵的注意力。</p>
 <p data-pid="YAnu4G5q">我们先看看 Windows 是怎么利用的：</p>
 <p data-pid="MbWzTz4P">1.（最大化窗口的）右上角：关闭，光标推过去之后再稍稍往左移动一点是退出最大化、最小化</p>
 <p data-pid="_-tKVLaY">2. 右下角：经典的“显示桌面”，而且把按钮直接精简到了一小条：（我敢说就因为显示桌面按钮在这，Windows 的桌面利用率比 macOS 高出许多）</p>
 <figure data-size="normal">
  <img src="https://picx.zhimg.com/50/v2-a2bc3afed1b12aca4a62becefaba1101_720w.jpg?source=1940ef5c" data-rawwidth="356" data-rawheight="111" data-size="normal" data-caption="" data-original-token="v2-389ea2f797a02c6e409aa1ad189cb2ab" data-default-watermark-src="https://picx.zhimg.com/50/v2-58948017099c24caa197338474f46cda_720w.jpg?source=1940ef5c" class="content_image" width="356">
 </figure>
 <p data-pid="nATiw_Yn">3. 下边缘：切换窗口（注意是<b>窗口 </b>“Window”，不是<b>程序 </b>“Application”），<b>类似</b>于 macOS 的 dock</p>
 <p data-pid="vy_eCQoJ">4. 左下角：经典的“开始菜单”，<b>类似</b>于 macOS 的启动台</p>
 <p data-pid="Tiq48M1J">更别说拖拽窗口情况下拖拽到屏幕任意角落 / 边缘可以调整窗口大小这种功能。</p>
 <p data-pid="tnxwOSm3">再来看看傻X macOS：</p>
 <ol>
  <li data-pid="gJ5j08ER">左上角：关于本机 / 系统设置 等等功能。嗯，macOS 用户日常工作一定会经常忘记自己电脑系统版本和硬件配置，所以才常常打开看看。对应于 Windows 的开始菜单，人家不仅同样能开关机、系统设置，还可以充当 macOS 聚焦搜索、启动台的功能</li>
  <li data-pid="2YLZ1WEt">上边缘：菜单。常用的菜单项一般都会记得快捷键，或者窗口中有专门的按钮（比如浏览器的加入书签之类的），所以菜单本身的定位就是收录用户偶尔才用的功能，反而你把它放在了最便捷的位置</li>
  <li data-pid="rPkiDWjP">右上角：今天 / 通知。虽然没有具体数据，但是我敢说频率肯定没有关闭窗口来的多。另外吐槽一下 macOS 3 个窗口按钮又圆又小真他妈难受。常用的功能交互区域就要大是一个基本的原则，抛去屏幕边缘操作不说，macOS 每次用都要用点注意力对准了点，Windows 的窗口按钮比这三个屎点儿不知道好按到哪里去了。</li>
  <li data-pid="1y2P7-GB">宝贵的左下右下角因为 dock 居中而完全浪费了。<br>
   这里再吐槽一下 dock 诡异的逻辑：是按照 application 来的而不是 window。有关 application 和 window 的从属逻辑我敢说除了工程师和深度用户一般人很难去理解的，因为用户能看到的只是方形的窗口，应用是隐藏在窗口后的逻辑。于是这里违背了“所见即所得”的基本交互原则：既然 dock 里的图标可以控制窗口，那么为什么图标和窗口不是一一对应？</li>
 </ol>
 <p data-pid="EO0Mf5Gz">简而言之，我认为 macOS 对屏幕角落<b>一个有效利用都没有</b>。当然这是窗口状态。如果是全屏状态呢？假设现在情景是：我在全屏浏览网页，想切换 tab 页，我来带你们见识一下这个傻屌系统：</p><a class="video-box" href="https://link.zhihu.com/?target=https%3A//www.zhihu.com/video/1015567732640595968" target="_blank" data-video-id="" data-video-playable="" data-name="" data-poster="https://pic3.zhimg.com/80/v2-ac68622a46b948c005a9cb8d53169712_b.jpg" data-lens-id="1015567732640595968"><img class="thumbnail" src="https://pic3.zhimg.com/80/v2-ac68622a46b948c005a9cb8d53169712_b.jpg"><span class="content"><span class="title"><span class="z-ico-extern-gray"></span><span class="z-ico-extern-blue"></span></span><span class="url"><span class="z-ico-video"></span>https://www.zhihu.com/video/1015567732640595968</span></span></a>
 <p data-pid="6GK75PVK">的确，firefox 的交互设计是一个方面，safari 在菜单下弹的时候就会自动向下平移，并且 tab 菜单在地址栏下面。先不说这种牺牲用户注意力而整体平移 UI 的做法是不是恰当的，重点是，macOS 不仅自己没利用好屏幕上边缘，而且<b>直接拒绝了其他应用在全屏状态下利用屏幕上边缘的可能</b>。一个鬼鬼祟祟总弹出的菜单栏把一切都毁了。</p>
 <p data-pid="PxAgrt_H">这样的缺点导致的问题是：做同样一种频繁操作，macOS 用户需要比 Windows 用户花费更多的精力去“瞄准”按钮。累计下来，就会让人感觉用 macOS 会很“累”，但又具体说不出原因。</p>
 <p data-pid="FsJKtBaC">你可以说，有关屏幕边缘的优化都通过触摸板手势和快捷键弥补了。但是这正是开始我说 macOS 的傲慢之处：你是想让用户做设计、打游戏也用你的触摸板，还是右手同时放着 Magic Trackpad 和 Magic Mouse，来回切换？作为用户，我可能习惯键鼠操作、纯键盘操作、touchpad 结合键盘操作等等，甚至是一手鼠标一手薯片的肥宅模式。在 Windows 下，不管怎样的方式，都可以有很好的解决方案；然而 macOS 却高度依赖于设备和快捷键。</p>

 <p data-pid="0_C6nXnB">经评论提醒，想起 macOS 还有触发角功能，上面的回答错误的评价了 macOS “没有对角落的利用”。为什么忘记这个功能呢？因为我用过，把右下角设置成显示桌面。然后呢，这个触发方式是全屏也有效的，打开的那个下午，当我退出视频的全屏模式（按钮正好在右下角），误触了好几次，然后就关掉了。：）换句话说这种必要的功能做成类似 hack 的方式本身就是错误的：设置路径深；设置后没有明显的 UI 提示；触发方式敏感容易误触。</p>
